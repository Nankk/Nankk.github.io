1 技術グループについて

1.1 概要

TRPGセッションをVRChat上で行うためのセットアップする際に、当初想定していなかった種々の問題が生じたことから個人での対応が困難となったため、Qボドゲ部情報強者の方々に縋るようにして作られたグループです。

1.2 活動内容

技術的に解決すべき課題が生じたときに随時その対応をすることが主な活動内容です。 割り振りは適当で、タスクはその場その場でやりたい人が引き受けたり、もしくは適当だと思われる人に依頼したりしてます。

1.3 進行中のプロジェクト

以下は現在取り組んでいる課題とそれに対する試みのまとめです。

1.3.1 進行用スライドの実装

  1. したいこと

    ボドゲ会本会（普段のオフラインの集まりをこう呼ぶことにします）で用いられるような資料表示用のスクリーンをVR空間内に配置し、進行役が任意のタイミングで任意の画像を映せるようにすることを目標とします。

  2. 考えられる実装手段

    スクリーンを設置して利用するためには、VRChatの仕様に沿った ワールド を作成し、その中にスクリーンの役割を担うオブジェクトを配置する必要があります。

    ワールド作成には3Dゲームエンジン「Unity」を使用します（VRChat指定のバージョンでないと正常に動作しないので注意）。 Unityで画像切り替えのようなギミックを実現する場合、通常はC#等でスクリプトを書いてオブジェクトの振る舞いを記述しますが、VRChatではセキュリティ上の理由から ユーザのカスタムスクリプトの動作を禁止する という クソみたいな 仕様があります。

    そのため、VRChat対応のギミックを作る際はVRChat公式により提供されたSDKを用い、限られたリソースを組み合わせながら工夫して作成する必要があります。

    スクリーンの実装に利用できそうなSDK部品には以下のようなものがあります。

    1. VRC_WebPanel
      • VRChatワールド内に外界の画像を持ち込むための最も有力な手段
      • …だったが脆弱性が見つかり2018年に死亡
      • 未だ復活の見通し立たず
    2. VRC_Panorama
      • 指定したURLの画像を表示するオブジェクト
      • URLの指定はワールド編集時にしか行えないため実行時の動的な画像指定が困難
        • 応答が動的に変わるURL（phpなど）を指定し、外部からAPIを通じて制御を行うのが旬
      • 不可軽減のためと思われるが、キャッシュを死ぬまで離さないので一工夫必要
    3. VRC_SyncVideoStream
      • 外部のストリーミング配信を移すことのできるオブジェクト
      • 遅延および解像度、参加者の回線速度等について要アセス
  3. 現時点でのソリューション

    これまでに色々と試したことについて以下に簡単にまとめます。

    1. VRC_Panorama

      VRC_Panoramaに任意の画像を表示するような柔軟性を持たせるためには、参照させるURLを一意に定め、URLが返す画像を任意のタイミングで変更させる仕組みをもたせる必要があります。 そこで現在「Heroku」上にスクリプトを配置し、クエリ文字列を指定することで画像の切り替えを行えるようにしています。

      以下はこの仕組みを利用しつつ、キャッシュによる表示の固定化をうまく回避して画像更新を行う方法を模索した履歴です。

      1. NextPano()による変更通知

        キャッシュを手放さないVRC_Panoramaに十分なサイズのキャッシュ配列を持たせ、全ての要素が埋まるまで強制的に新規読み込みをさせる方法です。ただし、キャッシュ配列が埋まった段階で画像更新ができなくなるという難点があります。 ユーザはワールド内に配置されたスイッチを押すことでこのイベントを発火させることができます。

        次のキャッシュ要素に進めるためにはVRC_PanoramaクラスのNextPano()メソッドのコールが必要となりますが、通常の方法ではUnityから同メソッドにアクセスすることができないため、 Animation および AnimatorController オブジェクトを介するテクニックを使用して無理矢理コールします。 詳細は別資料に譲りますが、VRChatでは頻出のテクニックです。

        現時点ではこの方法が比較的安定して動作しています（ユーザ間で反映タイミングのばらつきはあり）。

      2. 破壊&再生成（スイッチ）

        そんなにキャッシュを手放したくないのならば死ね！（過激派） ということでVRC_Panoramaをキャッシュごと破壊し、新たなVRC_Panoramaに新規読み込みを託す方法です（生成段階ではキャッシュを持たないため必ずデータを取りに行く）。

        何がしかの理由により後から入ったユーザから見える画面が乱れる現象が発生し、期日までに修正できなかったため一旦保留としています。 ちなみに2人目のみ鏡を通すと正常な画面を見ることができるという謎現象が確認されています（3人目からは直視・鏡のいずれでも表示が乱れる）。

        メモ：DestroyObjectがユーザ間で上手く同期されず複数スライドが重なって表示されていた？ObjectSyncを実装していなかった気がするのでそのせいかも

      3. 破壊&再生成（タイマー）

        VRC_Panoramaインスタンスにタイマーを持たせ、自身の破壊と生成を再帰的かつ自動的に行っていく方法です。 破壊&再生成の際に生じる読み込みラグにより画面が点滅するため、描画用のパネルを2枚用意しそれらを交互に表示することでちらつきを抑制します（OpenGLでいうSwapBuffers()のイメージ）。

        VRC_Panoramaを抹殺することでキャッシュも破壊されるはずでしたが、死んだVRC_Panoramaのキャッシュに対し何故かGCが働かず、溜まったゴミがPCのリソースを看過できないレベルで圧迫したので実戦投入は危険と判断されました。回避策を検討中です。

    2. VRC_SyncVideoStream（進行中）
      1. via Twitch配信

        スライドに映したい内容をTwitchで配信し、それをVRC_SyncVideoStreamで映し出す方法です。 Twitchへの反映7秒、その後VRChatが読み込むまでのラグを足して30秒ほどの遅延が生じたため実戦投入は厳しいと判断されました。

      2. via 独自サーバ？

        Twitchがダメなら自作してしまえ！な方法。現在進行中。
